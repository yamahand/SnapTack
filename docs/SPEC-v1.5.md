# SnapTack 仕様書 追補 (v1.5) — スクラップリスト

本書は SPEC.md (v1.0) / SPEC-v1.x.md (v1.1〜v1.4) への追補である。既存の仕様は特記なき限り維持する。

> SPEC.md 6章では「スクラップリスト」「ゴミ箱」を v2.0+ に分類していたが、v1.5 として実装する。SPEC.md の該当表も更新済み。

## 1. 目的と背景

現状、付箋は**画面に貼るだけ**で、以下の不便がある。

| 課題 | 内容 |
|---|---|
| 見失う | 付箋が増えると他ウィンドウに隠れる。どこにあるか分からなくなる |
| 誤って閉じると失われる | 中クリックで即座に消え、復元できない |
| 再起動で消える | アプリを終了すると全て失われる。長期間参照したい情報に使えない |

v1.5 では**スクラップリスト**を導入し、付箋を「画面上の存在」から「管理対象のデータ」へ格上げする。

コンセプトの「情報の一時保持」は維持しつつ、**一時保持の期間をセッションからアプリの再起動をまたぐ範囲へ広げる**。

## 2. 機能仕様

### 2.1 スクラップリストの概念

スクラップは以下の 3 状態を持つ。

| 状態 | 意味 | 画面表示 | リスト表示 |
|---|---|---|---|
| `Pinned` | 画面に貼られている | あり | あり |
| `Stashed` | リストにのみ存在する(画面から退避) | なし | あり |
| `Trashed` | 閉じた(ゴミ箱) | なし | ゴミ箱タブのみ |

- キャプチャで作られたスクラップは `Pinned` で始まる(現行と同じ体験)。
- 付箋を閉じる (中クリック等) と **`Trashed` へ移る。即座には削除しない**(現行は破棄)。
- `Trashed` から復元すると `Pinned` に戻る。

### 2.2 一覧 UI

**専用ウィンドウ(サムネイル一覧)を採用する。**

検討した選択肢:

| 案 | 利点 | 欠点 | 判断 |
|---|---|---|---|
| **専用ウィンドウ** | サムネイルで内容を判別できる。並べ替え・複数選択・ゴミ箱タブを載せられる | 実装量が最も多い | **採用** |
| トレイメニューに列挙 | 実装が軽い | サムネイルを出せず、テキストだけでは内容を判別できない。項目数が増えると破綻する | 不採用 |
| 付箋を一列に並べる | 追加ウィンドウ不要 | 画面を占有する。付箋が多いと成立しない | 不採用 |

**スクラップは画像であり、テキストのラベルでは判別できない**ことが決め手。サムネイル表示が必須要件になる。

#### 仕様

- トレイメニューの「スクラップリスト(&L)」または `Ctrl+Shift+L`(既定、変更可)で開く。
- 通常のウィンドウ(タスクバーに表示する、リサイズ可、Topmost ではない)。
- 二重に開かない(既に開いていればアクティブ化する。設定ウィンドウと同じ方式)。
- タブまたはフィルタで「スクラップ」/「ゴミ箱」を切り替える。
- 各項目に表示するもの:
  - サムネイル(既定 96×96 DIP、アスペクト比維持・中央クロップ。サイコロ表示と同じ方式)
  - キャプチャ日時
  - サイズ(物理px、`{0} × {1}` 形式)
  - 状態アイコン(`Pinned` / `Stashed` の区別)
- 空の場合はその旨のメッセージを表示する(無言の空白にしない)。

#### 操作

| 操作 | 動作 |
|---|---|
| ダブルクリック | `Pinned` にして画面に出し、その付箋を最前面へ持ってくる |
| `Enter` | 同上 |
| 右クリック | コンテキストメニュー |
| `Delete` | ゴミ箱へ移す(ゴミ箱内なら完全削除・確認あり) |
| `Ctrl+C` | 選択中スクラップの画像をコピー |

コンテキストメニュー: 「画面に表示 / 画面から隠す」「コピー」「PNGで保存...」「ゴミ箱へ移動」(ゴミ箱内では「復元」「完全に削除」)。

- 複数選択に対応する(`Ctrl` / `Shift` クリック)。上記操作は選択中の全件に適用する。

### 2.3 付箋側の変更

- コンテキストメニューに「リストに隠す」を追加する(`Stashed` へ移す = ウィンドウを閉じるがデータは残す)。
- **中クリック / 「閉じる」の挙動を変更する**: 破棄ではなく `Trashed` へ移す。
  - 誤操作で失われる問題を解消する。ユーザーから見た即時の挙動(消える)は変わらない。
- 付箋ウィンドウは**スクラップ 1 件に対して最大 1 つ**。既に表示中のものを再度「画面に表示」した場合は、新規生成せず既存ウィンドウをアクティブ化する。

### 2.4 永続化

**アプリ終了後もスクラップを保持し、次回起動時に復元する。**

#### 保存先

設定ファイル (`settings.json`) と同じ規則に従う。

```
<設定と同じディレクトリ>/scraps/
  index.json          … メタデータ (一覧)
  <id>.png            … 画像本体
```

- `id` は GUID。ファイル名衝突と、外部から中身を推測されることを避ける。
- ポータブル運用のため exe 同階層を優先し、書き込み不可なら `%APPDATA%\SnapTack` へフォールバックする(`SettingsStore` と同じ判定を再利用する)。

#### `index.json` のスキーマ

| キー | 型 | 内容 |
|---|---|---|
| `Id` | string (GUID) | 画像ファイル名と対応 |
| `CapturedAt` | string (ISO 8601) | キャプチャ日時 |
| `State` | string | `Pinned` / `Stashed` / `Trashed` |
| `TrashedAt` | string? | ゴミ箱へ移した日時(自動削除の判定に使う) |
| `PhysicalRect` | {X,Y,Width,Height} | キャプチャ元の位置・サイズ(物理px) |
| `OpacityPercent` | int | 不透明度 |
| `IsDice` | bool | サイコロ状態 |
| `WindowPosition` | {X,Y}? | 最後に表示していた位置(物理px)。null なら `PhysicalRect` の位置 |

#### 保存のタイミング

- スクラップの追加・状態変更・削除時に `index.json` を書き直す。
- 画像は追加時に 1 度だけ書き、以後変更しない(不透明度・サイコロ状態は元画像に影響しないため)。
- **付箋の移動のたびに保存しない**。ドラッグ中の書き込みを避けるため、位置は「ウィンドウを閉じる時」と「アプリ終了時」に保存する。

#### 復元

- 起動時に `index.json` を読み、`Pinned` のスクラップを画面に復元する。
  - 保存時のモニタ構成と異なる場合(モニタが減った等)、画面外に出る付箋は表示中のモニタ内へクランプする(サイコロ復元時と同じ処理を再利用)。
- `index.json` が壊れている・読めない場合は、**空のリストとして起動する**(現行の設定ファイルと同じ方針。アプリは落とさない)。
- 画像ファイルが欠けているエントリは読み飛ばし、`index.json` から取り除く。

#### 容量の管理

無制限に溜まると使用量が問題になるため、上限を設ける。

| 項目 | 既定値 | 設定可否 |
|---|---|---|
| スクラップの最大保持数 (`Pinned` + `Stashed`) | 200 件 | 可 |
| ゴミ箱の最大保持数 | 50 件 | 可 |
| ゴミ箱の自動削除 | 30 日経過で削除 | 可 (0 で無効) |

- 上限を超えた場合、古いものから削除する(`Pinned` は削除対象にしない)。
- 削除時は画像ファイルも消す。

### 2.5 設定項目の追加

| キー | 内容 | 既定 |
|---|---|---|
| `ScrapListHotkey` | スクラップリストを開くホットキー | `Ctrl+Shift+L` |
| `MaxScraps` | 最大保持数 | 200 |
| `MaxTrashedScraps` | ゴミ箱の最大保持数 | 50 |
| `TrashRetentionDays` | ゴミ箱の保持日数 (0 で無期限) | 30 |
| `RestoreScrapsOnStartup` | 起動時に `Pinned` を復元するか | true |

### 2.6 終了時の挙動

SPEC.md 4.6 の「開いている付箋はすべて閉じる」は維持するが、**閉じる = 破棄ではなく状態を保存する**に変わる。次回起動時に `Pinned` のものが復元される。

## 3. 設計上の変更

### 3.1 付箋の自己完結設計を解消する

現行は `ScrapWindow` を生成後 App 側で参照を保持していない(CLAUDE.md に明記)。スクラップリストからの操作には**中央管理が必須**のため、この設計を変更する。

- `ScrapManager` (仮) を導入し、スクラップのコレクションと永続化を担当させる。
- `ScrapWindow` は「`ScrapItem` を表示するビュー」に位置づけを変える。ウィンドウの生成・破棄は `ScrapManager` が行う。
- **これにより CLAUDE.md の「App から付箋を一括操作する機能を足さないこと」という制約は解除される。**同ファイルの記述も更新する。

### 3.2 影響範囲

| ファイル | 変更内容 |
|---|---|
| `Models/ScrapItem.cs` (新規) | スクラップ 1 件のモデル(状態・メタデータ) |
| `Models/ScrapManager.cs` (新規) | コレクション管理・永続化・ウィンドウのライフサイクル |
| `Models/ScrapStore.cs` (新規) | `index.json` と画像の読み書き |
| `Views/ScrapListWindow.xaml` (新規) | 一覧 UI |
| `Views/ScrapWindow.xaml.cs` | `ScrapItem` を受け取る形へ。閉じる = `Trashed` へ |
| `App.xaml.cs` | `ScrapManager` の生成、起動時復元、終了時保存 |
| `TrayIcon.cs` | メニューに「スクラップリスト」を追加 |
| `Models/AppSettings.cs` | 2.5 の設定項目を追加 |
| `Resources/Strings.resx` (+ `.ja`) | 追加する UI 文字列 |

### 3.3 リソース管理の注意

- 一覧に多数のサムネイルを出すため、**元画像をそのまま `Image.Source` にしない**。`BitmapImage.DecodePixelWidth` でデコード時に縮小し、メモリ使用量を抑える。
- 復元時に全画像を一度に読み込むと起動が遅くなる。`Pinned` の画像のみ起動時に読み、`Stashed` / `Trashed` のサムネイルは**リストを開いた時に遅延読み込みする**。
- `BitmapSource` は従来どおり `Freeze()` する。

## 4. 非機能要件

- **起動時間**: スクラップが上限 (200件) 存在しても、常駐開始までの体感を損なわない(1 秒未満目安)。上記の遅延読み込みで担保する。
- **既存設定との互換性**: v1.4 以前の `settings.json` をそのまま読めること。追加キーは既定値で補う。
- **`scraps/` が無い状態からの起動**: 初回起動と同じ扱いで正常に動作すること。

## 5. スコープ外 (v1.5 では実装しない)

| 項目 | 理由 |
|---|---|
| スクラップのタグ・検索・並べ替え | まず基本機能を固める。件数が増えてから必要性を判断する |
| スタイル機能(枠線・影の変更) | 独立した機能であり、リストとは別に検討する |
| モニタまたぎキャプチャ | 混在 DPI での正規化方針の検討が必要。SPEC-v1.x 2.4 の判断を維持する |
| クラウド同期・エクスポート | 「一時保持」というコンセプトから外れる |
| スクラップの編集(トリミング等) | v2.0+ で検討する |

## 6. 未決事項

実装着手前に判断が必要な点。

1. **`Stashed` と `Trashed` をユーザーがどう理解するか** — 「隠す」と「閉じる」の違いが伝わるか。UI 文言で解決できるか、状態を 2 つに減らすべきか、実装時に再検討する。
2. **画像の保存形式** — PNG (可逆・大きい) で確定してよいか。200 件保持すると数百 MB になり得る。容量が問題になるなら WebP 等を検討するが、`System.Windows.Media.Imaging` の標準エンコーダに WebP は無く、外部パッケージ追加の判断が必要になる(CLAUDE.md の「外部 NuGet は原則追加しない」に抵触)。
3. **ゴミ箱の自動削除を実行するタイミング** — 起動時のみか、リストを開いた時も行うか。
