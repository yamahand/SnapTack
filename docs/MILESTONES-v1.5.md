# SnapTack マイルストーン (v1.5) — スクラップリスト

MILESTONES-v1.x.md (M7〜M12) の続き。詳細仕様は SPEC-v1.5.md を参照。

M13 から M17 まで順に実装し、M17 完了時点で v1.5 をリリースできる。
**依存関係があるため上から順に実装すること**(M13 が全ての土台になる)。

## M13: スクラップのモデル化と中央管理

**ゴール: 付箋が `ScrapManager` 経由で生成・管理される(見た目と操作は現行のまま)**

これは**挙動を変えないリファクタリング**である。ユーザーから見た動作は v1.4 と同一になること。

- [ ] `Models/ScrapItem.cs` — スクラップ 1 件のモデル(Id / 画像 / 状態 / メタデータ)
- [ ] `Models/ScrapManager.cs` — コレクション管理とウィンドウのライフサイクル
- [ ] `ScrapWindow` を `ScrapItem` を受け取る形へ変更する
- [ ] `App.xaml.cs` で `ScrapManager` を生成し、`SelectionCompleted` から `ScrapManager.Add()` を呼ぶ
- [ ] 付箋を閉じたときに `ScrapManager` へ通知が届くこと(この時点では従来どおり破棄でよい)
- [ ] CLAUDE.md の「App から付箋を一括操作する機能を足さないこと」の記述を更新する

**確認**: v1.4 と同じ操作感で動作する(キャプチャ→付箋→移動→コピー→閉じる)。複数枚同時に出せる。

> この時点では永続化も一覧 UI も無い。**先に土台だけを差し替えて、既存の挙動が壊れていないことを確認する**のが狙い。

## M16: 永続化

**ゴール: アプリを再起動してもスクラップが残っている**

- [ ] `Models/ScrapStore.cs` — `scraps/index.json` と画像の読み書き
- [ ] 保存先は `SettingsStore` と同じ規則(exe 同階層 → `%APPDATA%` フォールバック)
- [ ] 追加・状態変更・削除時に `index.json` を更新する

## 実装順の補足

- **M13 は挙動を変えないリファクタリング**。ここでロジックを変えると、以降の不具合の切り分けが困難になる。
- M14 と M15 は分けているが、M14 単体では UI が無く確認しづらい。**M14 の確認はテストコードで行うのが現実的**。
- M16 の永続化は最後から 2 番目に置いた。メモリ上の状態管理 (M14) が固まる前にディスク形式を決めると、スキーマ変更の手戻りが発生するため。
- **M15 のホットキーは `GlobalHotkey` の拡張が前提**。この拡張を M15 の最初のタスクとして行う(既存のキャプチャホットキーを壊さないため)。
- 未決事項は SPEC-v1.5.md 6章にまとめてある。状態数は 3 状態で決定済み。**M13 着手前に残るのは画像の保存形式 (PNG で進める案) の最終判断のみ**。
